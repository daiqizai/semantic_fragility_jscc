# AI 协作规则

本项目可能由多个相互独立、无法共享聊天上下文的 AI 对话共同开发。
本地项目文件是唯一可信的共享记忆。

## 文档职责

- `README.md`：安装、运行方法和项目结构。
- `PROJECT.md`：研究目标、方法定义和不可随意改变的约定。
- `PROGRESS.md`：精简记录当前阶段、结果摘要、风险和下一步。
- `EXPERIMENTS.md`：所有实验、命令、配置、结果和产物的索引。
- `outputs/EXP-xxx/`：完整日志、配置快照、逐项指标和可视化。

禁止把完整训练日志、逐 epoch 指标或大段终端输出写入 `PROGRESS.md`。

## 每次开始任务

1. 依次阅读：
   - `AGENTS.md`
   - `PROJECT.md`
   - `PROGRESS.md`
   - `EXPERIMENTS.md`
   - `README.md`
2. 检查 Git 状态和最近提交。
3. 不得假设其他对话做过但未记录的工作。
4. 确认当前阶段、现有代码和下一项最高优先级任务。

## 每次结束任务

必须更新 `PROGRESS.md`，包括：

- 本次完成内容；
- 修改的文件；
- 执行的命令；
- 测试或实验结果；
- 新发现的问题；
- 下一步建议。

以上内容只写摘要和产物路径。涉及实验时，还必须更新
`EXPERIMENTS.md`，完整日志保存在对应 `outputs/EXP-xxx/`。

## 工作状态

任务只能标记为：

- `TODO`：尚未开始；
- `IN_PROGRESS`：正在进行；
- `BLOCKED`：无法继续，并注明原因；
- `DONE`：已经实现且完成验证；
- `INVALID`：实验或方案已被否定。

没有运行验证，不得标记为 `DONE`。`STALE` 只用于标记实验结果的有效性，
不作为任务执行状态。

## 实验规范

每个实验使用唯一 ID：

```text
EXP-S{阶段}-{序号}
```

例如：

```text
EXP-S1-001
EXP-S2-003
```

每次实验必须记录：

- 日期；
- Git commit；
- Git 工作区是否干净；
- 数据集及划分；
- 模型和 checkpoint；
- 完整配置；
- 随机种子；
- 运行命令；
- 日志及输出路径；
- 指标结果；
- 成功、失败或中断状态；
- 结果解释。

禁止覆盖旧实验目录或选择性删除失败结果。每次重跑必须使用新实验 ID。

## 多对话并发规则

1. 修改文件前先检查 Git 状态。
2. 不得还原或覆盖其他对话留下的修改。
3. 尽量让每个对话处理独立模块。
4. 公共接口发生变化时，必须更新 `PROJECT.md` 和 `PROGRESS.md`。
5. 发现不明修改时，先理解并保留，不得擅自删除。
6. 同一时间避免多个对话修改同一文件。
7. 长时间训练必须记录命令、PID、日志和输出目录。
8. 修改评估逻辑后，标记所有受影响的旧实验为 `STALE`。
9. 关键结论必须能追溯到实验 ID。
10. 文档记录与代码不一致时，以实际代码和可验证产物为准，并立即修正文档。

## 研究真实性

- 不得虚构实验结果；
- 不得将计划写成已实现；
- 不得将单次偶然结果写成稳定结论；
- 不得隐藏失败实验；
- 不得在不同预算、数据划分或评估条件下进行不公平比较；
- 所有论文结论必须能追溯到代码、配置、日志和结果。
